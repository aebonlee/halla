2일차 · 프롬프트 엔지니어링 & API 기초

같은 모델도 프롬프트 설계와 파라미터에 따라 결과가 달라집니다. 코드에서 직접 Claude API를 호출해 봅니다.

오늘의 시간표

오전 · 1~3교시
개념 + 라이브 데모 — 시스템 프롬프트와 메시지 구조, 파라미터의 의미
점심
점심 시간
오후 · 5~9교시
실습 + 빌드 — Cursor에서 Claude API를 직접 호출하고 출력 형식 다듬기

학습 목표

  • 시스템 프롬프트로 AI의 역할·말투·규칙을 고정할 수 있다.
  • 코드에서 Claude API를 호출하고 응답을 받아 화면에 쓸 수 있다.
  • temperature·max_tokens 등 파라미터가 결과에 주는 영향을 설명할 수 있다.
  • JSON 등 구조화된 형식으로 답을 받아 프로그램에서 처리할 수 있다.

시스템 프롬프트 설계

같은 모델도 시스템 프롬프트에 따라 완전히 다른 비서가 됩니다. 역할·맥락·출력 형식 — 어제 배운 SCORE의 연장입니다.

📝 좋은 시스템 프롬프트 4요소
① 너는 누구인가(역할) · ② 무엇을 돕는가(목표) · ③ 어떻게 말하는가(말투/길이) · ④ 하지 말아야 할 것(가드레일)

메시지 구조

  • system — AI의 정체성·규칙 (대화 내내 유지)
  • user — 사용자가 보낸 말
  • assistant — AI가 답한 말 (대화 이력으로 다시 전달)

API로 호출하기

핵심 흐름은 어떤 언어든 똑같습니다: 키 준비 → 메시지 구성 → 호출 → 응답 처리.

// 개념 흐름 (실제 코드는 강사와 함께 작성)
요청 = {
  model: "claude-opus-4-8",
  system: "너는 한라대 학생을 돕는 친절한 학습 도우미야.",
  messages: [ { role: "user", content: "오늘 배운 내용 3줄 요약해줘" } ],
  max_tokens: 500
}
응답 = AI에게_보내기(요청)
화면에_출력(응답.text)
⚠️ 키는 코드에 직접 넣지 않기
API 키는 환경변수로 불러옵니다. 깃허브에 올라가는 코드에 키 문자열을 박지 않습니다.

파라미터 다루기

  • temperature — 0에 가까우면 일관·딱딱, 1에 가까우면 다양·창의. 사실 답변은 낮게, 아이디어는 높게.
  • max_tokens — 답변 최대 길이. 너무 작으면 잘리고, 크면 비용↑.
  • stop / 시스템 규칙 — 형식을 강제하거나 특정 지점에서 멈추기.

실습

같은 질문을 temperature 0 / 0.7 / 1.0으로 3번 호출해 결과를 비교합니다.

구조화된 출력

프로그램이 답을 "읽으려면" 형식이 일정해야 합니다. 자유 문장 대신 JSON 같은 구조로 받습니다.

// "제목, 요약, 태그 3개를 JSON으로 줘" 라고 형식을 지정
{
  "title": "...",
  "summary": "...",
  "tags": ["...", "...", "..."]
}
💡 왜 중요한가
구조화 출력이 되면 AI의 답을 다른 기능에 자동으로 연결할 수 있습니다 — 내일 챗봇·모레 RAG의 토대.

오늘의 과제

  1. 나만의 시스템 프롬프트 1개 작성 (역할·말투·가드레일 포함)
  2. 같은 질문을 temperature 3종으로 호출해 차이 캡처
  3. "제목·요약·태그"를 JSON으로 받는 호출 성공시키기
