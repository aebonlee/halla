학습 목표

  • 임베딩이 "의미를 숫자 벡터로" 바꾸는 것임을 설명할 수 있다.
  • 문서를 로드·분할·임베딩해 벡터스토어(Chroma)에 저장할 수 있다.
  • 질문과 유사한 문서를 검색(retriever)하고 첫 RAG 체인을 만들 수 있다.
  • 코사인 유사도의 원리를 이해하고, chunk_size·k 등을 튜닝할 수 있다.

임베딩이란

임베딩(embedding)은 텍스트를 의미가 담긴 숫자 벡터로 바꾸는 것입니다. 의미가 비슷한 문장은 벡터도 가깝습니다.

  • "강아지"와 "개" → 벡터가 가까움
  • "강아지"와 "주식 시장" → 벡터가 멂
  • 가까움/멂은 코사인 유사도로 잰다
💡 왜 필요한가
컴퓨터는 글자를 "이해"하지 못합니다. 의미를 숫자로 바꿔야 "비슷한 문서 찾기"(검색)가 가능 → RAG의 핵심 부품.

임베딩 호출해 보기

OpenAI 임베딩 모델로 문장을 벡터로 바꿉니다. 벡터의 "차원 수"는 모델마다 다릅니다 (lang_06 원본).

from openai import OpenAI
client = OpenAI()

response = client.embeddings.create(
    input="Hello~ I'm Dr.Will",
    model="text-embedding-3-small",
)
vec = response.data[0].embedding
print(vec[:5])        # 벡터 앞 5개 값 (실수들)
print(len(vec))       # 차원 수 (예: 1536) — 모델마다 다름
📝 참고
모든 임베딩 모델이 1536차원은 아닙니다. 모델별 차원은 OpenAI 임베딩 문서에서 확인.

문서 로드

검색 대상 문서를 불러옵니다. 예제로 미국 국정연설 텍스트를 내려받습니다 (lang_07 원본).

import urllib.request
urllib.request.urlretrieve(
    "https://raw.githubusercontent.com/hwchase17/chat-your-data/master/state_of_the_union.txt",
    filename="state_of_the_union.txt",
)

from langchain_community.document_loaders import TextLoader
raw_documents = TextLoader("state_of_the_union.txt").load()
print(type(raw_documents), len(raw_documents))   # Document 리스트

청크로 분할

긴 문서는 통째로 다루기 어렵습니다. 청크(chunk)로 잘게 나눕니다 — "1000자짜리 상자"에 담는다고 생각하세요.

from langchain_text_splitters import CharacterTextSplitter

# chunk_size: 한 상자에 담을 글자 수 / chunk_overlap: 상자끼리 겹치는 글자 수
text_splitter = CharacterTextSplitter(chunk_size=1000, chunk_overlap=0)
documents = text_splitter.split_documents(raw_documents)

print("청크(상자) 개수:", len(documents))
print(documents[0].page_content[:200])   # 첫 상자 내용 일부
💡 overlap이 왜 있나
문장이 상자 경계에서 잘리면 의미가 끊깁니다. 약간 겹치게(overlap) 하면 경계 정보 손실을 줄입니다.

벡터스토어 구축

각 청크를 임베딩해 Chroma 벡터스토어에 저장합니다. 청크 1개 → 벡터 1개.

from langchain_openai import OpenAIEmbeddings
from langchain_chroma import Chroma

# 청크들을 임베딩해서 벡터DB에 저장 (상자마다 1536차원 벡터 부여)
db = Chroma.from_documents(documents, OpenAIEmbeddings())
# 예: 42개 상자 → 42개 벡터 (42 × 1536개의 수치)

유사도 검색

질문을 벡터로 바꿔, 가장 가까운 청크들을 찾아옵니다 (기본 4개).

query = "What did the president say about Ketanji Brown Jackson"
docs = db.similarity_search(query)      # k=4 기본
print(docs[0].page_content)

# 벡터로 직접 검색하기 (질문→임베딩→유사도)
qvec = OpenAIEmbeddings().embed_query(query)
docs2 = db.similarity_search_by_vector(qvec)
print(len(qvec))   # 질문 벡터 차원

검색은 어떻게 동작하나

전체 흐름을 한 장으로 정리하면:

문서
 → 1000자 상자(chunk)로 분할  →  예: 42개 상자
 → 각 상자를 임베딩          →  상자마다 1536차원 벡터
 → 벡터스토어(Chroma)에 저장

[질문]
 → 질문도 1536차원 벡터(QV)로 변환
 → QV와 42개 상자 벡터의 유사도(코사인) 계산
 → 가장 가까운 상자 k개 반환
📝 감각
RAG는 AI를 "더 똑똑하게" 만드는 게 아니라 "좋은 자료를 손에 쥐여 주는" 것. 상자 분할·임베딩 품질이 답 품질을 좌우합니다.

retriever

체인에서 검색을 담당하는 부품이 retriever입니다. "질문을 주면 유사 문서 k개를 주는" 인터페이스.

# k=3: 가장 유사한 상자 3개만 가져오는 검색기
retriever = db.as_retriever(search_kwargs={"k": 3})
hits = retriever.invoke("대통령이 말한 핵심은?")
for d in hits:
    print("-", d.page_content[:80])

첫 RAG 체인

검색된 문서를 프롬프트의 context로 넣어 답하게 하면 RAG(검색 → 삽입 → 생성)입니다 (lang_07 원본).

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

chat_model = ChatOpenAI(model="gpt-4o", temperature=0)
prompt = ChatPromptTemplate.from_template("""
다음 문서를 참고해서 질문에 답변하세요.

문서:
{context}

질문:
{question}
""")

def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)   # 여러 청크를 한 덩어리로

chain = (
    {
        "context":  lambda q: format_docs(retriever.invoke(q)),  # 질문→검색→합치기
        "question": lambda q: q,
    }
    | prompt | chat_model | StrOutputParser()
)

print(chain.invoke("케탄지 브라운 잭슨에 대해 뭐라고 했어? 한국어로 답해줘."))

format_docs

검색된 여러 청크(Document)를 모델이 읽기 좋은 하나의 문자열로 합치는 도우미입니다. 출처를 함께 넣을 수도 있습니다(내일 활용).

def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

# 확장: 출처(metadata)도 함께
def format_docs_with_meta(docs):
    return "\n\n".join(f"{d.page_content}\n(출처: {d.metadata})" for d in docs)

직접 구현 — 코사인 유사도

블랙박스를 열어, 임베딩·벡터스토어를 직접 만들어 보면 원리가 또렷해집니다 (lang_08 개념).

import numpy as np
from sklearn.metrics.pairwise import cosine_similarity

# 두 문장이 의미적으로 얼마나 가까운지 = 코사인 유사도(1에 가까울수록 유사)
def embed(text):
    return client.embeddings.create(input=text, model="text-embedding-3-small").data[0].embedding

a = np.array(embed("강아지가 공을 물고 뛴다")).reshape(1, -1)
b = np.array(embed("개가 공놀이를 한다")).reshape(1, -1)
c = np.array(embed("내일 주식 시장 전망")).reshape(1, -1)
print("강아지↔개:", cosine_similarity(a, b)[0][0])   # 높음
print("강아지↔주식:", cosine_similarity(a, c)[0][0]) # 낮음
💡 SimpleVectorStore
lang_08은 SimpleTextLoader·SimpleVectorStore·SimpleRetriever를 직접 구현합니다. "저장된 벡터들과 질문 벡터의 유사도를 재서 가장 가까운 것을 반환"이 전부입니다.

한글 임베딩 & 튜닝

한국어 문서는 한국어에 강한 임베딩이 유리할 수 있습니다. 또 chunk_size·k를 바꿔 검색 품질을 조정합니다.

# 한글 친화 임베딩 (HuggingFace) — OpenAI 임베딩과 품질 비교
from langchain_huggingface import HuggingFaceEmbeddings
emb_ko = HuggingFaceEmbeddings(model_name="jhgan/ko-sroberta-multitask")

# 튜닝 포인트
# - chunk_size: 작게(500) → 정밀하지만 맥락 적음 / 크게(1500) → 맥락 많지만 노이즈↑
# - chunk_overlap: 100~200 권장
# - k: 너무 작으면 근거 부족, 너무 크면 노이즈/비용↑
📝 튜닝 실험
같은 질문에 chunk_size 500 vs 1500, k 2 vs 5를 비교해 "어떤 설정이 내 문서에 맞는지" 직접 찾으세요.

오늘의 실습 미션

  1. 내 문서(txt)를 로드·분할·임베딩해 벡터스토어 만들기
  2. 질문 5개로 유사도 검색 → 잘 찾는지 확인
  3. retriever + RAG 체인으로 내 문서 기반 답변 받기
  4. chunk_size 500 vs 1500 비교 — 답이 어떻게 달라지나
  5. 코사인 유사도로 "비슷한 문장 / 다른 문장" 점수 비교

자주 막히는 부분

  • chromadb 설치 오류 → !pip install -U langchain-chroma chromadb 후 런타임 재시작
  • 검색이 엉뚱함 → chunk가 너무 크거나, 질문이 모호. chunk_size·k 조정
  • 한글 검색이 약함 → 한국어 임베딩(HuggingFace) 시도
  • 비용/속도 → 임베딩은 문서가 많을수록 비용↑. 필요한 문서만 넣기
🤖 막히면
검색 결과(retriever.invoke(q))를 먼저 출력해 "근거 문서가 제대로 잡히는지"부터 확인하세요. 답이 이상하면 대개 검색 단계 문제입니다.
